# melon — Claude Code Context

## Project Overview

Melon is a dependency manager for agentic markdown — skills, agents, and prompts for AI coding tools. It resolves GitHub-hosted skills, fetches them via sparse git checkout, pins them in a lock file, and symlinks them into agent tool directories (`.claude/skills/`, `.windsurf/skills/`, etc.).

Two binaries are produced from the same code: `melon` and `mln` (alias).

## Common Commands

```sh
# Build and install both binaries locally
make install          # runs: go install ./cmd/mln && go install ./cmd/melon

# Run tests
go test ./...

# Run a specific package's tests
go test ./internal/resolver/...

# Run tests with verbose output
go test -v ./...
```

## Architecture

```
cmd/melon, cmd/mln        — entry points; read version from build info, call cli.Run()
internal/cli/             — cobra commands: init, install, add, remove, list, search, info
internal/manifest/        — melon.yaml parsing and writing
internal/lockfile/        — melon.lock parsing, writing, and diffing
internal/resolver/        — transitive dependency resolution (BFS, greedy highest-compatible)
internal/fetcher/         — sparse git checkout, tree hashing, version resolution
internal/placer/          — symlink creation/removal into agent directories
internal/store/           — .melon/ cache directory management
internal/agents/          — hardcoded agent → skill directory conventions table
internal/index/           — melon-index.yaml search (curated skill registry)
internal/github/          — thin GitHub REST API client
pkg/semver/               — semver constraint checking (^, ~, exact)
openspec/                 — BDD-style specs and archived change proposals
```

## Key Patterns

**Install pipeline** (in order): Resolve → Fetch → Write lock → Place symlinks → Prune removed deps

**Idempotency:** Fetch skips if tree hash matches. Place removes existing symlink before recreating. Install can be re-run safely.

**Dependency injection for testability:** `resolver.Resolve()` accepts `resolveVersion` and `fetchManifest` function params so tests can provide fakes without network access.

**TTY detection:** Bubbletea TUI (spinners, progress bars, interactive selects) is only shown when stdout is a terminal. CI/scripting gets plain text output automatically.

**Sparse checkout:** Each dep is a shallow clone with only the needed subdirectory checked out. Results are copied to `.melon/<dep@version>/` and symlinked from there.

**Tree hashing:** SHA-256 over sorted file paths + contents (not git tree objects). Used to detect if a cached dep is stale.

**Error handling:** Sentinel errors (`ErrConflict`, `ErrNoSemverTags`, `ErrUnknownAgent`) wrapped with `fmt.Errorf("%w", err)`. No panics in normal operation.

**Determinism:** Maps are sorted before iteration. Lock diffs are in stable order. Tree hashes are deterministic.

## Important Notes

- **GitHub-only:** Dependency paths are GitHub paths (`github.com/owner/repo/...`). No GitLab/Gitea support.
- **Go 1.25+** is required (see `go.mod`). Uses recent language features.
- **`CGO_ENABLED=0`** — pure Go, no C deps.
- **Git required at runtime:** Fetcher shells out to `git` via `exec.Command`. Modern git with sparse-checkout support is needed.
- **`GITHUB_TOKEN`** env var: Used by the resolver and GitHub client for higher API rate limits. Not required but recommended.
- **Agent conventions** are hardcoded in `internal/agents/agents.go`. Adding a new AI tool means adding an entry there.

## melon.yaml Schema

```yaml
name: string
version: string          # semver
description: string
entrypoint: string       # e.g. "SKILL.md"
dependencies:
  github.com/owner/repo/path: "^1.0.0"  # semver constraint OR branch name
tool_compat:
  - claude-code          # drives where skills are placed
outputs:                 # optional: overrides tool_compat placement
  .claude/skills/: "*"
tags: []
```

## melon.lock Schema

Generated by `melon install`. Commit this file. Contains exact version, git tag, repo URL, subdir, entrypoint, SHA-256 tree hash, and file list per dependency.

## Specs

Design decisions and requirements live in `openspec/specs/`. Each spec is a BDD-style markdown file. Read the relevant spec before making changes to a subsystem. Archived changes (proposals + tasks) are in `openspec/changes/archive/`.

## Release Process

Releases are triggered by pushing a `v*` tag. GoReleaser builds `melon` and `mln` binaries for darwin/linux/windows (amd64 + arm64). A follow-up GitHub Actions job publishes the npm wrapper packages to npm with platform-specific optionalDependencies.
